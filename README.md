# 🤖 AI Review Bot — мобильное приложение для Android (без бэкенда)

Нативное приложение на **Kotlin** для **Android**: вся работа выполняется **только в телефоне**. Своего сервера (бэкенда) нет — приложение само забирает отзывы из Wildberries, классифицирует их, при необходимости строит контекст (RAG), генерирует ответы через Yandex GPT и отправляет ответы обратно в Wildberries. Все данные и логика хранятся и выполняются на устройстве.

---

## 🎯 Принцип: без бэкенда

- **Нет своего сервера** — не нужен Python-бэкенд, хостинг, Docker.
- **Внешние сервисы** — приложение с телефона обращается только к официальным API:
  - **Wildberries API** — получение отзывов и отправка ответов (токен/ключ продавца пользователь вводит в настройках приложения).
  - **Yandex Cloud API** — генерация ответов (Yandex GPT) и при необходимости эмбеддинги для RAG (ключ и folder_id пользователь вводит в приложении).
- **Всё остальное в телефоне:** база отзывов и архив для RAG, классификация, решение «автоотправка или модерация», очередь модерации, статистика — локально (Room, логика на Kotlin).

---

## ✨ Что делает приложение (всё на устройстве)

- **Загрузка отзывов** — приложение само запрашивает новые отзывы в Wildberries API (по токену из настроек).
- **Классификация** — на устройстве: оценка сложности, проверка по blacklist, категория отзыва (правила или лёгкая ML-модель в приложении).
- **RAG (похожие отзывы)** — локальный архив отзывов в приложении; поиск похожих по эмбеддингам (Yandex Embeddings API с телефона или локальная модель) и хранение векторов/текста в Room или файле.
- **Генерация ответа** — запрос к Yandex GPT API с телефона с контекстом (текст отзыва + при необходимости RAG).
- **Решение** — на устройстве: автоотправка или отправка в очередь модерации по порогам (сложность, уверенность, рейтинг, blacklist).
- **Модерация** — экраны в приложении: просмотр очереди, одобрить / отредактировать / отклонить; отправка выбранного ответа в Wildberries API с телефона.
- **Статистика и настройки** — локальная аналитика (обработано, автоответы, ошибки), настройки порогов и blacklist хранятся на устройстве.

Никакой промежуточный сервер не используется — только телефон и внешние API (WB, Yandex).

- **Нет своего бэкенда** — только приложение и два внешних API.
- **Логика в приложении:** классификация, RAG, принятие решений, очередь модерации — реализованы на Kotlin и работают на устройстве.
- **Сеть нужна** только для вызовов Wildberries и Yandex; без сети можно просматривать кэш и очередь, но не загружать новые отзывы и не генерировать ответы.

## 📚 Отличие от варианта «с бэкендом»

| Аспект              | С бэкендом (старый вариант) | Без бэкенда (текущий)        |
|---------------------|-----------------------------|------------------------------|
| Сервер              | Свой Python-сервер          | Нет                          |
| Откуда отзывы       | Бэкенд дергает WB API       | Приложение дергает WB API    |
| Классификация / RAG | На сервере                  | В приложении (Kotlin + Room) |
| Генерация ответов   | На сервере (Yandex GPT)     | С телефона → Yandex API      |
| Модерация           | Telegram или веб            | Только в приложении          |
| Данные              | БД на сервере               | Room и настройки на устройстве |

Итог: **вся работа делается только в телефоне**; приложение — единственное место, где выполняется логика и хранятся данные (плюс вызовы WB и Yandex по сети).

### Ошибка Yandex: «folder ID does not match with service account folder ID»

Если в логах или в приложении появляется ошибка **400** с текстом вроде «Specified folder ID '…' does not match with service account folder ID '…'», значит **каталог (Folder ID)** в настройках приложения не совпадает с каталогом, в котором создан API-ключ. В настройках приложения в поле **Folder ID** нужно указать **тот же каталог**, к которому привязан ключ (тот, что в сообщении об ошибке указан как «service account folder ID»). После исправления запросы к Yandex GPT должны проходить.

## 💡 Идеи развития продукта (brainstorm-ideas-new)

Этот раздел — результат первичного brainstorm-а для **нового продукта** (Android‑приложение без бэкенда), но на базе уже спроектированного репозитория. Идеи сгруппированы по трём ролям (Product Manager, Product Designer, Software Engineer), а затем выделены **5 приоритетных направлений** для следующих итераций.

### 👩‍💼 Перспектива Product Manager (ценность и рынок)

- **1. Поддержка нескольких магазинов/аккаунтов WB в одном приложении**  
  Для агентств и продавцов с несколькими кабинетами: быстое переключение профилей, суммарная и помагазинная статистика.
- **2. Режим «консервативный/агрессивный» автоответов**  
  Простая настройка стратегии: в консервативном режиме больше уходит в модерацию, в агрессивном — больше автоответов (с понятными пределами риска).
- **3. Шаблоны ответов и бренд‑тональность**  
  Набор базовых шаблонов (вежливый, экспертный, дружелюбный) + свои заготовки, чтобы ответы ИИ были «в стиле бренда», а не обезличенными.
- **4. Простая «экономика использования»**  
  Экран, который показывает: сколько отзывов обработано, сколько времени сэкономлено и сколько приблизительно стоили бы те же ответы руками (ценность для пользователя).
- **5. Сценарий для саппорт‑команд**  
  Роль «оператор поддержки»: акцент на очереди модерации, совместная работа нескольких операторов (в перспективе — синхронизация через общий аккаунт WB).

### 🎨 Перспектива Product Designer (опыт и вовлечение)

- **1. Онбординг‑мастер при первом запуске**  
  Пошаговый мастер, который помогает ввести токены, объясняет ограничения и показывает «тур по ключевым экранам» перед боевым использованием.
- **2. Понятная визуализация решения «авто/модерация»**  
  Значки/цвета и короткие подписи, объясняющие почему отзыв попал в автоответ или в модерацию (доверие к ИИ, меньше страха у пользователя).
- **3. Дружелюбные пустые состояния и демо‑режим**  
  Если ещё нет отзывов или токенов — показывать демо‑данные/скринсторы, а не пустой список (помогает новичкам понять ценность приложения до настройки всех ключей).
- **4. Быстрые действия со списка отзывов**  
  Свайпы/иконки для интуитивных действий: «в модерацию», «разобраться позже», «открыть детали», чтобы меньше переходить по экранам.
- **5. Экран «здоровье аккаунта»**  
  Простая панель: доля отвеченных отзывов, доля негативных, среднее время ответа, «смайлик здоровья» — мотивация поддерживать высокий уровень сервиса.

### 👨‍💻 Перспектива Software Engineer (техника и платформа)

- **1. Фоновая синхронизация отзывов (WorkManager)**  
  Периодическая загрузка новых отзывов и постановка их в очередь даже когда приложение не открыто (с учётом ограничений Android и энергопотребления).
- **2. Локальное кеширование промптов и ответов для отладки**  
  Включаемый «developer mode»: храним последние запросы/ответы Yandex GPT для отладки и улучшения промптов (безопасно и локально).
- **3. Плагинная архитектура движка классификации**  
  Интерфейс, позволяющий легко переключаться между «правилами», «он‑девайс моделью» и «гибридным» подходом, не переписывая весь пайплайн.
- **4. Тонкая телеметрия (локальная) по времени обработки**  
  Сбор метрик по каждому шагу пайплайна (WB → классификация → RAG → GPT → отправка), чтобы видеть узкие места без сторонних сервисов аналитики.
- **5. Расширяемый формат локального архива для будущего RAG**  
  Проработка схемы хранения, которая позволит безболезненно включить полноценный RAG в v2 (индексы, версии схемы, миграции Room).

### ⭐ Топ‑5 приоритетных идей (что делать в первую очередь)

Ниже — 5 идей, которые особенно важны для **первых версий продукта**, потому что они:  
1) усиливают основную ценность (быстро и безопасно помогать с отзывами);  
2) дают возможность быстро проверить гипотезы;  
3) создают задел для дальнейшего развития.

1. **Онбординг‑мастер при первом запуске (Designer #1)**  
   - **Почему важно:** без хорошего онбординга начинающий пользователь легко теряется на этапе ввода токенов и не видит ценность приложения.  
   - **Что проверить:**  
     - Пользователь понимает, какие ключи/токены нужны и где их взять.  
     - После онбординга большинство пользователей доходят до первого успешного ответа на отзыв.  
   - **Предположения:**  
     - Целевая аудитория готова потратить 2–3 минуты на пошаговую настройку.  
     - Подробные подсказки (линки на документацию WB/Yandex) снизят процент «брошенных» установок.

2. **Понятная визуализация решения «авто/модерация» (Designer #2 + PM #2)**  
   - **Почему важно:** доверие к автоответам критично — пользователь должен понимать, что происходит и почему.  
   - **Что проверить:**  
     - Пользователь по одному взгляду на карточку отзыва понимает, что с ним сделано и почему.  
     - После добавления визуализации уменьшается количество отключений автоответов.  
   - **Предположения:**  
     - Короткие текстовые пояснения + иконки достаточно, чтобы объяснить логику без чтения документации.  
     - Пользователь хочет видеть причины решения, но не в формате «стены текста».

3. **Шаблоны ответов и бренд‑тональность (PM #3)**  
   - **Почему важно:** ИИ‑ответы должны звучать как «бренд», а не как обезличенный бот; это ключ к долгосрочному использованию.  
   - **Что проверить:**  
     - Пользователи действительно настраивают свои шаблоны и выбирают тональность.  
     - Шаблоны сокращают время правок перед отправкой ответа.  
   - **Предположения:**  
     - У пользователей уже есть представление о «голосе бренда», которое можно выразить в 2–3 параметрах.  
     - Простой интерфейс выбора тональности (переключатель/пресеты) даёт ощутимую пользу без сложных настроек.

4. **Фоновая синхронизация отзывов (Engineer #1)**  
   - **Почему важно:** ценность приложения растёт, когда пользователь открывает его и сразу видит свежие отзывы, а не запускает ручную загрузку.  
   - **Что проверить:**  
     - Фоновая синхронизация стабильно работает в реальных условиях (ограничения Android, батарея, сеть).  
     - Пользователь воспринимает регулярные уведомления о новых отзывах как полезные, а не как спам.  
   - **Предположения:**  
     - Объём данных отзывов сравнительно небольшой и не приведёт к серьёзной нагрузке на батарею.  
     - Пользователи не против периодического фона при прозрачной настройке частоты/времени.

5. **Экран «здоровье аккаунта» (Designer #5 + PM #4)**  
   - **Почему важно:** даёт простую, визуальную метрику прогресса и ценности (сколько отзывов закрыто, насколько вовремя отвечаем).  
   - **Что проверить:**  
     - Пользователь регулярно заходит на этот экран и использует его как ориентир.  
     - Появление «здоровья аккаунта» увеличивает долю обработанных отзывов.  
   - **Предположения:**  
     - Несколько простых показателей (доля отвеченных, среднее время ответа, доля негатива) достаточно для понятной картины.  
     - Геймификация (смайлики/цвета) мотивирует пользователей поддерживать высокий уровень сервиса.
