# Локальная логика на устройстве (Фаза 0.3)

**Назначение:** описание правил классификации отзывов, движка решений (автоотправка / модерация) и плана по RAG. Всё выполняется в приложении на Kotlin, без бэкенда.

---

## 1. Классификация отзывов (первая версия)

### 1.1 Выбор подхода для MVP

- **Вариант A (выбран для v1):** правила на Kotlin — ключевые слова, blacklist, простая оценка сложности по рейтингу и длине текста. Не требует ML-модели, быстро реализуется, предсказуемо.
- **Вариант B (на будущее):** лёгкая on-device модель (например TensorFlow Lite) для категории/сложности — больше точность, но сложнее интеграция и размер приложения. Оценить во второй итерации.

### 1.2 Входы классификатора

- Текст отзыва (`reviewText: String`).
- Рейтинг отзыва (`rating: Int`, обычно 1–5).

### 1.3 Выходы классификатора

- **category** — категория отзыва (например: `gratitude`, `complaint`, `question`, `neutral`). Определяется по ключевым словам (спасибо, отлично → gratitude; брак, возврат, плохо → complaint и т.д.).
- **complexityScore** — оценка сложности от 1 до 5:
  - 1 — простой (короткий положительный, благодарность);
  - 5 — сложный (длинный, негативный, жалоба, вопросы по возврату/гарантии).
  - Правила: низкий рейтинг (1–2) повышает сложность; длинный текст повышает; наличие слов из «сложных» списков повышает.
- **blacklistTriggered** — `true`, если в тексте встретилось хотя бы одно слово/фраза из списка blacklist (возврат, брак, юрист, суд, претензия, Роспотребнадзор и т.д.). Список настраивается пользователем в настройках и хранится локально.

### 1.4 Правила (без кода, для реализации в Kotlin)

1. Blacklist проверять первым: если есть совпадение (без учёта регистра) → `blacklistTriggered = true`, `complexityScore = 5`.
2. Рейтинг 1–2 → базовая сложность не ниже 3; рейтинг 5 + короткий текст + слова благодарности → сложность 1.
3. Длина текста: очень короткий (например &lt; 20 символов) и рейтинг 4–5 → снижать сложность; длинный (например &gt; 200 символов) → повышать.
4. Ключевые слова для категории: свои списки под gratitude, complaint, question; при отсутствии совпадений — `neutral`.
5. Пороги и списки слов (в т.ч. blacklist) брать из локальных настроек (DataStore/Room), не хардкодить.

## 2. Движок решений (автоотправка или модерация)

### 2.1 Входы

- Результат классификации: `complexityScore`, `blacklistTriggered`, при необходимости `category`.
- Оценка уверенности генератора (`confidenceScore`, 0.0–1.0) — если Yandex GPT или постобработка её возвращают; иначе можно не использовать или считать 1.0 для простых кейсов.
- Рейтинг отзыва (`rating`).
- Настройки пользователя: порог сложности, порог уверенности, минимальный рейтинг для автоответа.

### 2.2 Выход

- **AUTO_SEND** — отправить сгенерированный ответ в WB сразу, без модерации.
- **MODERATE** — записать в очередь модерации, пользователь сам одобрит/отредактирует/отклонит.

### 2.3 Правила (по порядку проверки)

1. **Blacklist:** если `blacklistTriggered == true` → всегда **MODERATE** (без исключений).
2. **Рейтинг:** если `rating < minRatingForAutoResponse` (настройка, например 3) → **MODERATE** (низкий рейтинг — только модерация).
3. **Сложность:** если `complexityScore > complexityThreshold` (настройка, например 4) → **MODERATE** (сложные отзывы не автоотправляем).
4. **Уверенность:** если используется `confidenceScore` и `confidenceScore < confidenceThreshold` (настройка, например 0.8) → **MODERATE**.
5. Во всех остальных случаях → **AUTO_SEND**.

Итог: сначала проверяем blacklist и рейтинг, затем сложность и уверенность; при любом срабатывании условия «на модерацию» — MODERATE, иначе AUTO_SEND.

## 3. RAG (поиск похожих отзывов) — не в MVP

### 3.1 Решение для первой версии

- В **v1 RAG не реализуем.** Генерация ответа — только по текущему отзыву + системному промпту.
- Упрощает сроки и объём работ; качество для многих отзывов достаточное.

### 3.2 План на вторую итерацию (v2)

- **Источник эмбеддингов:** вызов Yandex Embeddings API с телефона или компактная on-device модель (оценить размер и скорость).
- **Хранение:** таблица в Room «архив отзывов»: id, текст отзыва, вектор эмбеддинга (массив Float или сериализованная строка/Blob). Либо хранение векторов в файле, индексирование при старте.
- **Поиск:** для текущего отзыва получить эмбеддинг → поиск K ближайших по косинусной близости в локальном хранилище → форматирование контекста (например, топ-5 похожих текстов) и передача в промпт Yandex GPT.
- **Заполнение архива:** при обработке/отправке ответов добавлять отзыв в архив; опционально — разовая выгрузка из WB (архив отзывов) и индексация. Детали — на этапе проектирования v2.

## 4. Итог по фаза 0.3

- **Классификация v1:** правила на Kotlin, blacklist из настроек, сложность 1–5, категория по ключевым словам.
- **Движок решений:** строгая последовательность проверок (blacklist → рейтинг → сложность → уверенность); пороги из настроек.
- **RAG:** не в MVP; схема для v2 кратко зафиксирована выше.

После реализации этих правил в коде приложение сможет полностью обрабатывать отзыв на устройстве без бэкенда.
