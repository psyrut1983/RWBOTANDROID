# UI/UX improvements (предложения) — RWBOTANDROID

Дата: 2026‑03‑17  
Область: Jetpack Compose (Material 3), навигация, экраны Reviews / Moderation / Stats / Settings / Review detail

## 1) Быстрый аудит текущего UI (что видно по коду)

### 1.1 Навигация и структура экранов
- Нижняя навигация (`NavigationBar`) есть, но **иконки пустые** (`icon = {}`) — хуже распознаваемость вкладок.
- Нет верхнего `TopAppBar` (контекста, действий, заголовков экранов) — пользователю сложнее понимать «где он».
- Экран модерации (`ModerationScreen`) фактически переиспользует `ReviewsScreen`, из‑за чего:
  - дублируется фильтр‑логика (там принудительно ставится `ON_MODERATION`, но `ReviewsScreen` всё равно рисует ряд фильтров),
  - появляется «двойной заголовок»/неясная иерархия (сначала текст “Очередь модерации”, потом внутри — UI обзора отзывов).

### 1.2 Типографика и масштабирование
- На экранах **массово захардкожены большие `sp`** (например 28/32/48sp).
  - Плюс: доступность для слабовидящих.
  - Минус: визуальная «перегруженность», разъезжающиеся карточки, меньше контента на экран, сложнее поддерживать единый стиль.
- Тема (`Typography`) почти пустая — Material‑типографика не используется как система.

### 1.3 Состояния (loading / empty / error / success)
- Loading сейчас главным образом «на кнопке» (“Загрузка…”) и `syncMessage` как просто текст.
- Нет явных пустых состояний:
  - “Ещё нет отзывов”
  - “Нет отзывов по фильтру”
- Ошибки показываются строкой без категории/действия (например “повторить”).

### 1.4 Формы настроек
- Поля токенов отображаются как обычный текст (нет маски/скрытия, нет кнопки “показать/скрыть”).
- Нет валидации формата и подсказок ошибок (например, “Folder ID слишком короткий”, “Токен WB пустой”).
- “Сохранено” — это текст на кнопке, но нет подтверждения уровня экрана (snackbar) и нет статуса “сохраняю…”.

### 1.5 Список отзывов и карточка
- Карточки кликабельны целиком, фото сверху — хорошо.
- Не хватает быстрых действий “сразу из списка” (например: “в модерацию”, “отклонить”, “копировать текст”, “поделиться”).
- Статус показан текстом (`${review.status}`), но нет визуального маркера (цвет/чип/иконка).

### 1.6 Экран детали отзыва
- Большой контент, но нет:
  - явной структуры “Отзыв / Ответ / Действия”,
  - поля редактирования ответа перед отправкой (важно для модерации),
  - подтверждения перед отправкой в WB (защита от случайных тапов),
  - состояния “идёт отправка/обработка” кроме блокировки кнопок.

### 1.7 Бейдж на иконке приложения
- Бейдж реализован через тихое уведомление — практично (на многих лаунчерах так и работает).
- Внизу навигации **нет бейджа на вкладке “Модерация”** (а это самый логичный UX‑якорь для неотвеченных/на модерации).

## 2) Предложения улучшений (что сделать в UI/UX)

Ниже — идеи, сгруппированные по приоритету. Для каждой — варианты реализации.

### A. Навигация и информационная архитектура (высокий приоритет)

#### A1. Добавить иконки в нижнюю навигацию
**Почему важно:** быстрее распознаётся вкладка, меньше ошибок навигации.  
**Варианты:**
- **Вариант 1 (быстро):** стандартные `Icons.Filled.*` из Material.
- **Вариант 2 (лучше):** кастомные иконки (если есть бренд‑гайд).

#### A2. Бейджи на вкладках (особенно “Модерация”)
**Почему важно:** бейдж на иконке приложения полезен, но внутри приложения важнее показать “где проблема”.  
**Варианты:**
- **Вариант 1:** `BadgedBox` на `NavigationBarItem` для “Модерация” (число `ON_MODERATION` или “неотвеченные”).
- **Вариант 2:** вместо числа — “точка” (dot) для минимализма.

#### A3. TopAppBar на каждом корневом экране
**Почему важно:** контекст + действия (“обновить”, “фильтр”, “поиск”, “настройки”).  
**Варианты:**
- **Вариант 1:** `Scaffold(topBar = { TopAppBar(...) })` в `MainNav` + переключение заголовка по `currentDestination`.
- **Вариант 2:** topBar внутри каждого экрана (гибче, но больше повторений).

### B. Состояния загрузки/ошибок/пустоты (высокий приоритет)

#### B1. Явные пустые состояния (Empty state)
**Что добавить:**
- “Нет отзывов — проверьте токен WB в настройках”
- “Нет отзывов по выбранному фильтру”
**Варианты:**
- **Вариант 1:** простая иллюстрация + текст + кнопка “Обновить”
- **Вариант 2:** “демо‑режим” с мок‑данными (для онбординга)

#### B2. Ошибки с категорией и действием
**Что улучшить:** вместо строки вверху — `Snackbar` + кнопка “Повторить”.  
**Распространённая ошибка новичков:** показывать ошибки “молча” в логах или одним Text без действия пользователя.

#### B3. Прогресс‑индикаторы не только в кнопке
**Варианты:**
- **Вариант 1:** `LinearProgressIndicator` сверху списка при синхронизации
- **Вариант 2:** skeleton‑карточки (приятнее, но сложнее)

### C. Типографика, дизайн‑система, доступность (высокий приоритет)

#### C1. Убрать хардкод `sp`, перейти на `MaterialTheme.typography`
**Почему важно:** единый стиль, меньше “ломается” на разных устройствах.  
**Варианты:**
- **Вариант 1 (консервативно):** сохранить крупные размеры, но определить их в `Typography` и использовать `style = ...`.
- **Вариант 2 (адаптивно):** сделать “Режим крупного текста” в настройках (normal/large/extra large).

#### C2. Минимальная визуальная иерархия в карточке
**Идея:** текст отзыва — `bodyLarge`, мета‑инфо (рейтинг/статус/артикулы) — `labelMedium`, статус — чип.

#### C3. Content descriptions для изображений/кнопок
Сейчас `contentDescription = null` у фото товара — для accessibility это “немой” элемент.  
**Вариант:** краткое описание вроде “Фото товара: артикул X”.

### D. Список отзывов: скорость работы и действия (средний приоритет)

#### D1. Поиск и сортировка
**Почему важно:** когда отзывов много, фильтров по статусу мало.  
**Варианты:**
- поиск по тексту/артикулу
- сортировка по дате/рейтингу/статусу

#### D2. Быстрые действия в карточке
**Варианты:**
- `SwipeToDismiss`/свайпы (быстро, но надо аккуратно с ошибками)
- кнопки‑иконки справа (менее “магично”, но понятнее)

### E. Модерация (высокий приоритет)

#### E1. Отдельный UX для “очереди модерации”
Сейчас это просто тот же список. Улучшить так:
- сверху: счётчик “на модерации”
- фильтры: “новые / просмотренные / с ошибкой отправки”
- батч‑действия: “одобрить все с автоответом” (опционально, опасно без подтверждения)

#### E2. Редактирование ответа перед отправкой
**Почему важно:** доверие и контроль.  
**Варианты:**
- текстовое поле на detail‑экране (мягкий старт)
- отдельный экран “редактор ответа” (чётче UX)

#### E3. Подтверждение перед отправкой в WB + undo
**Варианты:**
- диалог “Отправить ответ?” (простое)
- snackbar “Отправлено” + “Отменить” (если API позволяет/если отправка не мгновенная)

### F. Настройки и онбординг (высокий приоритет)

#### F1. Онбординг‑мастер первого запуска
**Шаги:** WB token → Yandex key → Folder ID → тестовый запрос → готово.  
**Варианты:**
- **Вариант 1:** внутри `SettingsScreen` (минимум навигации)
- **Вариант 2:** отдельные экраны “мастер” (лучший UX)

#### F2. Безопасность ввода токенов
**Что добавить:**
- скрытый ввод (password)
- кнопка “показать”
- “скопировано” после копирования
**Распространённая ошибка новичков:** хранить/показывать токены как обычный текст без маски и без предупреждений.

#### F3. Валидация и подсказки
Пример: если токены пустые — показать предупреждение “Синхронизация не будет работать”.

### G. Статистика (средний приоритет)

#### G1. Превратить “четыре числа” в понятную панель
**Варианты:**
- карточка “Сегодня / Неделя / Всё время”
- график тренда (потом)
- метрика “время ответа” (ценность)

## 4) Как тестировать (чек‑лист ручной проверки)

### Навигация
- Открыть приложение → проверить, что **у каждой вкладки есть иконка** и понятный заголовок сверху.
- Переключиться между вкладками 10–15 раз → убедиться, что состояние сохраняется (список/скролл/фильтр).

### Синхронизация и состояния
- На экране “Отзывы” нажать “Обновить”:
  - видно прогресс (индикатор),
  - кнопка блокируется,
  - при успехе — понятное сообщение,
  - при ошибке — snackbar + “Повторить”.
- Отключить интернет → проверить UX ошибки сети.

### Пустые состояния
- Очистить локальную БД (или поставить фильтр, который не совпадает ни с одним отзывом) → увидеть friendly empty state.

### Настройки
- Ввести токены → убедиться, что они **скрыты**, можно показать/скрыть.
- Оставить обязательные поля пустыми → увидеть подсказку/предупреждение.
- Нажать “Сохранить” → увидеть “сохраняю…” и подтверждение сохранения.

### Модерация и отправка
- Открыть отзыв со статусом `ON_MODERATION`:
  - есть понятная зона редактирования ответа,
  - перед “Отправить” появляется подтверждение,
  - после — явное подтверждение успеха/ошибки.

## 5) Примечания по рискам и типичным ошибкам

- **Не делайте “свайп удаляет/отправляет” без подтверждения**: пользователи часто промахиваются.
- **Не используйте только цвет** для статуса (дальтонизм) — добавляйте текст/иконку.
- **Не хардкодьте размеры** по всему UI — иначе сложно будет поддерживать и адаптировать.
